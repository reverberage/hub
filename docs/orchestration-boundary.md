# Orchestration Boundary

How opencode and n3rverberage divide responsibility. Not a user guide — this is the architectural
contract between the two systems. Future agents and developers should reference this before
modifying either side.

## 1. Conceptual Model

n3rverberage wraps opencode, not the other way around. n3rverberage is the **scaffolding and
infrastructure layer** that bootstraps, configures, and extends opencode. opencode is the
**agent runtime** — it hosts LLM interaction, executes subagents, and loads skills.

| Layer | Owned By | What It Does |
|-------|----------|-------------|
| **Agent runtime** | opencode | LLM interaction, subagent spawning (`task` tool), skill loading, command dispatch |
| **Scaffolding & init** | n3rverberage | `n3rverberage init` generates ALL opencode config files from Jinja2 templates: AGENTS.md, opencode.json, skills, commands, agents, plugins |
| **Task delegation** | n3rverberage A2A hub | Cross-process task routing by `skill_id`. HTTP server at `127.0.0.1:19820`. Persists tasks, routes to registered agents. |
| **Memory persistence** | n3rverberage MAGI | ChromaDB vector store + SQLite relations. SDD artifacts saved via topic keys (`sdd-<change_id>-<phase>`). |
| **Provider abstraction** | n3rverberage | `get_provider()` factory for chat LLMs (QwenProvider, FallbackProvider, OpenAIProvider, LocalProvider). `get_tts_provider()` factory for speech synthesis (TTSProvider — wraps Qwen-TTS HTTP API, NOT a ModelProvider subclass). Used by satellites, not by opencode. |
| **Model rotation** | hub scripts | `scripts/qwen_fallback.py` writes `opencode.json` on quota exhaustion. Rotates opencode's model. |
| **Config generation** | n3rverberage init | Every file under `.opencode/` is generated. Hub tracks only hub-owned files (docs, scripts, protocol specs). |

**Key rule**: if a file is generated by `n3rverberage init`, it is not tracked in hub. Exceptions: `scripts/qwen_fallback.py` and `scripts/scaffold-satellite.py` are hub-owned scripts that operate on n3rverberage-generated files.

## 2. Coordination Mechanisms

Two systems for dispatching work. They are NOT interchangeable.

### `task` tool — opencode (in-process)

Spawns a subagent within the same opencode session. The subagent inherits memory access,
file permissions, and tool availability from the parent.

**Use when:**
- Sequential SDD phases (1-5, 7-8). Each phase needs the prior phase's memory artifacts.
- Work requires immediate context from the current session.
- The subagent needs read/write access to the same filesystem.
- Latency matters — no network round-trip.

**Examples**: `sdd-explorer`, `sdd-speccer`, `sdd-verifier`, `sdd-archiver`

### `delegate_task` — n3rverberage A2A hub (cross-process)

Submits a task to the A2A hub. The hub persists it, matches it to an agent by `skill_id`, and
waits for that agent to poll and execute. The caller polls for completion.

**Use when:**
- Offloading work to a different process or machine (other agent instances).
- Blind adversarial review (judgment-day). The delegate must not see the local reviewer's findings.
- Long-running tasks that might outlive the current opencode session.
- The task requires a different model or agent capability not available locally.

**Examples**: Phase 6 implementation (`skill_id="implementation"`), judgment-day delegated review (`skill_id="review"`)

### Phase 6: Both Mechanisms

The SDD apply phase uses `delegate_task` first (luxury: another agent does the implementation),
then falls back to local `task` tool after 5 polls (~50s). This is the only phase that
crosses the boundary. The orchestrator prefers delegation but doesn't depend on it.

## 3. Memory Systems

Two memory stores coexist. The SDD pipeline uses MAGI exclusively.

| System | Store | Scope | Used By |
|--------|-------|-------|---------|
| **opencode built-in** | opencode internal | Session | Session context, compaction, agent state, short-term working memory |
| **n3rverberage MAGI** | ChromaDB + SQLite | Project-local persistent | SDD artifacts (topic keys), session summaries, agent verdicts, long-term institutional memory |

**MAGI is the source of truth for SDD state.** Every phase saves its artifact to MAGI via
`memory_save(topic_key="sdd-<change_id>-<phase>")`. Subsequent phases read from MAGI via
`memory_recall(topic_key=...)`.

The `n3rverberage-lifecycle.js` plugin bridges the two:
- On session compaction, injects current MAGI SDD state into compaction context
- On session idle (5 min), auto-archives a session summary to MAGI

## 4. Provider Layers

Three provider domains operate on the same API family (DashScope). Each uses a different
API endpoint and authentication scope.

| Layer | Config | Consumer | Interface | Purpose |
|-------|--------|----------|-----------|---------|
| **opencode's LLM** | `opencode.json` | The agent itself | OpenAI-compatible chat (`/v1/chat/completions`) | Controls which model the agent uses for thinking |
| **Satellite LLM** | `N3RVERBERAGE_PROVIDER` env | Satellite engines (`engine.py`) | `ModelProvider` Protocol: `complete()`, `complete_structured()`, `complete_with_tools()` | Controls which model satellites use via `get_provider()` |
| **Satellite TTS** | `N3RVERBERAGE_TTS_BASE_URL` env | Satellite engines (`engine.py`) | `TTSProvider.synthesize()` — standalone class, NOT a `ModelProvider` | Speech synthesis via Qwen-TTS HTTP API (`/api/v1/services/aigc/multimodal-generation/generation`). Model: `qwen3-tts-flash`. Voice: `Cherry`. Works with `sk-ws-*` workspace keys. Returns WAV audio. |

The LLM layers share the same 103M-token free quota pool. TTS uses a separate quota
(10,000 characters per model). Rotation systems operate at different levels:

- **`scripts/qwen_fallback.py`**: rotates opencode's model. Reads `opencode.template.json` (generated by `n3rverberage init`), substitutes `{model}`, writes `opencode.json`. Manual trigger: `--rotate`.
- **`FallbackProvider`**: rotates at satellite runtime. Chains chat providers. Catches `QuotaExhaustedError` and moves to the next. Automatic within a single satellite process.
- **TTS has no fallback chain**: TTS quota is per-character, not per-token. Exhaustion raises `TTSQuotaExhaustedError` directly.

### API Endpoints

All providers default to the legacy `dashscope-intl.aliyuncs.com` base URL. Alibaba recommends
migrating to workspace-specific domains for better performance and stability:

| Service | Legacy | Recommended |
|---------|--------|-------------|
| Chat LLM (OpenAI-compatible) | `https://dashscope-intl.aliyuncs.com/compatible-mode/v1` | `https://{WorkspaceId}.ap-southeast-1.maas.aliyuncs.com/compatible-mode/v1` |
| TTS (Qwen-TTS, non-realtime) | `https://dashscope-intl.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation` | `https://{WorkspaceId}.ap-southeast-1.maas.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation` |

Set via env vars: `N3RVERBERAGE_QWEN_BASE_URL` (chat + multimodal),
`N3RVERBERAGE_TTS_BASE_URL` (TTS). The legacy domain remains fully functional but
is no longer recommended for new projects.

**TTS key compatibility**: The old `services/audio/tts/SpeechSynthesizer` endpoint (CosyVoice)
does NOT accept `sk-ws-*` workspace API keys — it returns `"does not support http call"`.
The `services/aigc/multimodal-generation/generation` endpoint (Qwen-TTS) works with any
DashScope key, including `sk-ws-*` workspace keys. `cosyvoice-*` models are WebSocket-only
and unavailable via the workspace's HTTP API.

Model rotation changes opencode's model. FallbackProvider changes the satellite's model. Same quota,
different consumers.

## 5. Lifecycle

The full path from project initialization to completed SDD change:

```
n3rverberage init
  │  Generates: opencode.json, .opencode/* (skills, commands, agents, plugins),
  │  AGENTS.md, .n3rverberage/*, .githooks/*
  │
  ▼
opencode starts
  │  Loads opencode.json → selects model (e.g., qwen/qwen3-coder-plus)
  │  Loads .opencode/commands/ → slash commands available
  │  Loads .opencode/agents/ → subagents registered
  │  Loads .opencode/plugins/ → lifecycle hooks active
  │
  ▼
/sdd-new "add user auth"
  │  Orchestrator derives change_id="add-user-auth"
  │
  ├── Phase 1: task(sdd-explorer) → memory_save(sdd-add-user-auth-context) to MAGI
  ├── Phase 2: task(sdd-proposer) → memory_save(sdd-add-user-auth-proposal)
  ├── Phase 3: task(sdd-speccer)  → memory_save(sdd-add-user-auth-spec)
  ├── Phase 4: task(sdd-designer) → memory_save(sdd-add-user-auth-design)
  ├── Phase 5: task(sdd-task-planner) → memory_save(sdd-add-user-auth-tasks)
  ├── Phase 6: delegate_task(implementation)
  │   │  A2A hub routes to available agent
  │   │  Polls 5× ~10s
  │   └── Fallback: task(sdd-apply) → memory_save(sdd-add-user-auth-impl)
  ├── Phase 7: task(sdd-verifier) → memory_save(sdd-add-user-auth-verify)
  └── Phase 8: task(sdd-archiver) → memory_save(sdd-add-user-auth-done)
  │
  ▼
/judgment-day add-user-auth
  │  Parallel execution:
  ├── delegate_task(review) → A2A hub → independent review
  │                            → memory_save(sdd-add-user-auth-review-delegated)
  └── Local review (concurrent) → memory_save(sdd-add-user-auth-review-local)
  │
  ▼
  Synthesis: compare findings → classify (CONFIRMED/SUSPECT/CONTRADICTION)
  Fix loop: address CRITICAL+WARNING findings → re-verify (max 2 iterations)
  │
  ▼
qwen_fallback.py --rotate  (when quota exhausted)
  │  Reads opencode.template.json
  │  Substitutes next available model
  │  Writes opencode.json
  │  opencode must restart to pick up new model
```

## 6. What Hub Tracks vs What n3rverberage Owns

A concrete reference for what goes where.

### Tracked in hub (hub-owned)

| File | Purpose |
|------|---------|
| `docs/*.md` | Architecture, protocol specs, roadmap, model catalog |
| `scripts/qwen_fallback.py` | Model rotation for opencode |
| `scripts/scaffold-satellite.py` | Satellite skeleton generator |
| `tests/*.py` | Hub-level tests (scaffold validation) |
| `.opencode/commands/*.md` | Slash command definitions |
| `.opencode/skills/*/SKILL.md` | Agent skill instructions |
| `.opencode/agents/*.md` | Subagent definitions |
| `README.md`, `LICENSE`, `SECURITY.md` | Project metadata |

### Generated by n3rverberage init (gitignored, never tracked)

| File | Purpose |
|------|---------|
| `opencode.json` | opencode's active model config |
| `opencode.template.json` | Template for qwen_fallback.py rotation |
| `AGENTS.md` | Project coding standards (generated from template) |
| `.opencode/plugins/*.js` | Lifecycle and env injection plugins |
| `.opencode/tools/*.ts` | Custom tool wrappers |
| `.opencode/package.json` | Node deps for tools/plugins |
| `.n3rverberage/*` | A2A config, hub state, memory storage |
| `.githooks/*` | Git hook scripts |
| `CONTRIBUTING.md`, `SECURITY.md` | Standard community docs |

**The rule**: hub owns the contract (specs, protocols, commands, skills, roadmap). n3rverberage owns the implementation of that contract (config generation, provider behavior, memory persistence, task routing). The bridge is the init process: n3rverberage reads hub's skills/commands/agents at init time, renders them through templates, and writes the runtime config that opencode consumes.
